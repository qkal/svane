# Kvale v1.1 ‚Äî Design Spec

**Date:** 2026-03-29
**Authors:** Kal (Complexia) + Claude (Anthropic)
**Status:** Approved ‚Äî ready for implementation

---

## Overview

Kvale v1.1 completes the data layer. v1.0 shipped a clean foundation ‚Äî fetch, cache, retry, stale-while-revalidate. v1.1 adds the features that real production apps need: mutations with optimistic updates, request deduplication, reactive keys, cache invalidation, direct cache manipulation, and cache garbage collection. The API shapes are verified against TanStack Query v5 and SWR.

---

## Scope

### In v1.1

14 features across two categories:

**From the v1.0 deferred list:**
1. Reactive keys
2. Request deduplication
3. `cache.mutate()`
4. `cache.invalidate(key)`
5. Exponential backoff

**New additions (verified gaps vs commercial libraries):**
6. `AbortController` ‚Äî cancel in-flight fetches on destroy or key change
7. `cache.prefetch()` ‚Äî preload cache without creating a reactive result
8. `cache.getQueryData()` / `cache.setQueryData()` ‚Äî direct cache read/write
9. `cache.cancelQuery()` ‚Äî abort in-flight request for a key (required for safe optimistic updates)
10. `keepPreviousData` ‚Äî show old data during key transitions instead of flashing to `loading`
11. `refetchOnReconnect` ‚Äî re-fetch stale queries when browser comes back online
12. `select` ‚Äî transform data before delivering to component; cache stores raw `T`
13. `CacheStore.entries()` ‚Äî public iterator for future devtools
14. `gcTime` ‚Äî prune inactive cache entries after a configurable timeout

### Deferred to v1.2

- **SSR / SvelteKit `load()` integration** ‚Äî using Kvale in `+page.server.ts` / `+page.ts`
- **Infinite / paginated queries** ‚Äî `cache.infiniteQuery()` with `fetchNextPage`, `hasNextPage`
- **Devtools package** ‚Äî cache inspector UI (`kvale/devtools`) ‚Äî `entries()` in v1.1 is the prep
- **Structural sharing** ‚Äî deep equality check to prevent re-renders when data is unchanged
- **Query observers** ‚Äî `cache.subscribe(key, callback)` public API

### Known v1.1 limitation

Without `gcTime` enforcing cleanup, long sessions with many unique keys could accumulate stale entries. `gcTime` (added in v1.1) addresses this ‚Äî but only for entries with no active subscribers.

### Key architectural decisions

**Deduplication lives in `CacheStore`.**
A `Map<string, Promise<unknown>>` tracks in-flight requests by serialized key. `QueryRunner.execute()` checks this before calling `fn()`. If a promise exists, it attaches to it. On resolve or reject, the entry is cleared. This is the only way multiple `QueryRunner` instances for the same key can share a promise ‚Äî the store is already the shared source of truth.

**Reactive keys live in the adapter.**
`QueryConfig.key` accepts `() => string | unknown[]`. The adapter reads the key getter inside `$effect`, registering it as a Svelte reactive dependency. When the key changes, the `$effect` re-runs: old runner is destroyed, new runner is created. A mutable `let currentRunner` reference inside the adapter ensures `refetch()` on the returned `QueryResult` always calls the current runner.

**`select` applied at runner layer, not cache layer.**
`CacheStore` always stores raw `T`. `QueryRunner` applies `select: (data: T) => U` before notifying subscribers. The cache is never contaminated with derived data.

**`gcTime` tied to subscriber count.**
When the last subscriber to a key unsubscribes (runner destroyed), `CacheStore` starts a `gcTime` timer for that key. If a new subscriber arrives before expiry, the timer is cancelled. On expiry, the entry is deleted. Entries created via `prefetch` or `setQueryData` with no active subscribers start the `gcTime` timer immediately upon creation.

**Mutations are independent of the query cache.**
`MutationRunner` does not store results in `CacheStore`. It manages its own reactive state. Optimistic updates are the user's responsibility via `setQueryData` in `onMutate` ‚Äî `MutationRunner` only threads the `onMutate` return value (context) into subsequent callbacks.

## API

### `CacheConfig` ‚Äî 2 new options

```ts
interface CacheConfig {
  staleTime: number;              // existing ‚Äî default: 30_000
  retry: number;                  // existing ‚Äî default: 1
  refetchOnWindowFocus: boolean;  // existing ‚Äî default: true
  persist?: Storage;              // existing ‚Äî default: undefined
  refetchOnReconnect: boolean;    // NEW ‚Äî default: true
  gcTime: number;                 // NEW ‚Äî default: 300_000 (5 min)
}
```

### `QueryConfig` ‚Äî 2 changed, 3 new options

```ts
interface QueryConfig<T, U = T> {
  key: string | unknown[] | (() => string | unknown[]); // CHANGED ‚Äî accepts getter
  fn: (signal: AbortSignal) => Promise<T>;              // CHANGED ‚Äî receives AbortSignal
  staleTime?: number;
  refetchInterval?: number;
  enabled?: boolean | (() => boolean);
  keepPreviousData?: boolean;   // NEW ‚Äî default: false
  select?: (data: T) => U;      // NEW ‚Äî transforms data before delivery, changes QueryResult<U>
}
```

> Note: `onSuccess`/`onError`/`onSettled` are intentionally absent from `QueryConfig`. TanStack Query v5 removed them from query hooks because with shared cache, multiple subscribers firing the same callback causes bugs (e.g. 3 toasts for 1 fetch). The Svelte-idiomatic alternative is `$effect(() => { if (result.status === 'success') ... })`.

### `MutationConfig<TData, TVariables, TContext>` ‚Äî new

```ts
interface MutationConfig<TData, TVariables, TContext = unknown> {
  fn: (variables: TVariables, signal: AbortSignal) => Promise<TData>;
  onMutate?: (variables: TVariables) => TContext | Promise<TContext>;
  onSuccess?: (data: TData, variables: TVariables, context: TContext) => void;
  onError?: (error: Error, variables: TVariables, context: TContext) => void;
  onSettled?: (
    data: TData | undefined,
    error: Error | null,
    variables: TVariables,
    context: TContext,
  ) => void;
}
```

### `MutationResult<TData, TVariables>` ‚Äî new

```ts
interface MutationResult<TData, TVariables> {
  readonly status: 'idle' | 'loading' | 'success' | 'error';
  readonly data: TData | undefined;
  readonly error: Error | null;
  mutate(variables: TVariables): Promise<void>;
  reset(): void;
}
```

### New `cache` methods

```ts
// Mutation
cache.mutate<TData, TVariables, TContext = unknown>(
  config: MutationConfig<TData, TVariables, TContext>
): MutationResult<TData, TVariables>

// Invalidation ‚Äî uses matchesKey() (array prefix): invalidate(['todos']) matches ['todos'],
// ['todos', 1], ['todos', 'active'], etc.
cache.invalidate(key: string | unknown[]): void

// Prefetch ‚Äî populates cache without creating a reactive result.
// No-ops if entry exists and is still fresh (not stale). Refetches if stale or missing.
cache.prefetch<T>(config: Pick<QueryConfig<T>, 'key' | 'fn' | 'staleTime'>): Promise<void>

// Direct cache access
cache.getQueryData<T>(key: string | unknown[]): T | undefined
cache.setQueryData<T>(key: string | unknown[], data: T | ((prev: T | undefined) => T)): void

// Cancel in-flight request ‚Äî call in onMutate before optimistic write
cache.cancelQuery(key: string | unknown[]): void
```

### `CacheStore` ‚Äî 1 new public method

```ts
store.entries(): IterableIterator<[string, CacheEntry]>
```

## Data Flows

### Reactive key change

```
key getter returns new value
  ‚Üí $effect re-runs (Svelte tracked the getter dependency)
  ‚Üí old runner: unsubscribe() + destroy()
  ‚Üí new runner created with new serialized key
  ‚Üí keepPreviousData=true: new runner initializes with old data, status='refreshing'
  ‚Üí keepPreviousData=false: new runner initializes with status='loading'
  ‚Üí runner.execute() ‚Üí fetch with new key
```

### Request deduplication

```
runner.execute()
  ‚Üí store.getInFlight(serializedKey)
  ‚îú‚îÄ hit: attach .then()/.catch() to existing promise ‚Äî no new fetch
  ‚îî‚îÄ miss: call fn(signal) ‚Üí store.setInFlight(serializedKey, promise)
           promise settles ‚Üí store.clearInFlight(serializedKey) ‚Üí notify subscribers
```

### Optimistic mutation

```
cache.mutate({ fn, onMutate, onError, onSettled })
  ‚Üí mutationResult.mutate(variables)
  ‚Üí onMutate(variables)         ‚Äî user calls setQueryData (optimistic write), returns context
  ‚Üí fn(variables, signal)       ‚Äî server call
  ‚îú‚îÄ success ‚Üí onSuccess(data, variables, context) ‚Üí onSettled(data, null, variables, context)
  ‚îî‚îÄ error   ‚Üí onError(err, variables, context)    ‚Äî user calls setQueryData (rollback)
               ‚Üí onSettled(undefined, err, variables, context)
```

### `gcTime` cleanup

```
runner.destroy()  OR  setQueryData/prefetch with no active subscribers
  ‚Üí subscriber count for key = 0
  ‚Üí CacheStore starts gcTime timer for key
  ‚îú‚îÄ new subscriber arrives before expiry ‚Üí timer cancelled
  ‚îî‚îÄ timer fires ‚Üí CacheStore.delete(key) ‚Üí entry removed from memory
```

### `invalidate`

```
cache.invalidate(['todos'])
  ‚Üí CacheStore finds all entries where key starts with '["todos"'
  ‚Üí sets entry.timestamp = 0 (isStale() ‚Üí true)
  ‚Üí notifies all subscribers for each matching key
  ‚Üí active QueryRunners re-execute (status ‚Üí 'refreshing' if data exists)
```

## Error Handling

| Scenario | Behaviour |
|---|---|
| `fn()` rejects | Exponential backoff: `min(1000 √ó 2‚Åø, 30_000)ms`. After retries: `status: 'error'`, `error` set. Data preserved if was `'refreshing'`. |
| Abort (destroy / key change) | In-flight `fn()` receives `signal.aborted = true`. `fetchWithRetry` checks `destroyed` guard after await ‚Äî result silently discarded. |
| `onMutate` throws | Mutation transitions to `'error'` immediately. `fn` is never called. |
| `enabled` getter throws | Treated as `enabled: false`. Query stays `'idle'`. Never propagates. |
| `gcTime` timer race | If new subscriber arrives between `destroy()` and timer fire, timer is cleared. No data lost. |
| `setQueryData` on missing key | Creates a fresh cache entry with `timestamp: Date.now()`. Subscribers notified. |

- `entries()` ‚Äî iterator yields all current entries

- `AbortSignal` passed to `fn`, aborted on `destroy()`
- `keepPreviousData` ‚Äî key change: `data` preserved, `status: 'refreshing'` (not `'loading'`)
- `select` ‚Äî raw `T` in cache; subscriber receives `U`

**`tests/core/mutation.test.ts`**
- `idle ‚Üí loading ‚Üí success` lifecycle, `status` and `data` correct at each step
- `onMutate` context threaded into `onSuccess`, `onError`, `onSettled`
- Optimistic update: `setQueryData` called before `fn`, rolled back on error via `context`
- `reset()` returns to `idle`, clears `data` and `error`

- `keepPreviousData`: component never sees `status: 'loading'` between key changes

- `cancelQuery` in `onMutate` prevents optimistic data being overwritten by in-flight refetch
